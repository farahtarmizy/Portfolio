Designing a dual-document selection with no reference to work from

A niche use case with no design precedent online. Three iterations, self-imposed constraints, and a final design where the layout itself does the explaining.

Role
Sole designer
Type
New feature
Status
Shipped to client
What gap analysis does

Compliance officers need to check whether their internal policies are aligned with regulatory requirements. The gap analysis feature automates this — a compliance officer selects one authority document (a regulatory requirement) and one internal document (their company's policy), and the system generates an analysis highlighting what's missing, what evidence exists, how to comply, and how critical each gap is.

The design challenge was the selection step: how do you let a user pick from two completely different document lists — simultaneously, on one screen — in a way that feels clear and controlled?

The constraint that shaped everything

When I searched for design references, I found nothing directly applicable. Most document selection patterns used sequential flows — pick from list A on one screen, then pick from list B on the next. That would have meant multiple pages, more clicks, and users losing sight of their first selection by the time they made their second.

I set three constraints for myself before designing: no multiple pages, both selections visible at the same time, and easy to undo either selection independently.

These weren't requirements from the brief — they were decisions I made based on what I knew about how compliance officers work. They're reviewing specific document pairs. They need to be confident both documents are the right ones before committing to an analysis.

Three iterations to get there
Iteration 1 — timeline steps
Step-by-step wizard. Broke the constraint immediately — multiple pages, users lost visibility of first selection by step two.
Iteration 2 — modal with stacked lists
Both lists in one modal, stacked vertically. Better — but the Analyse button sat at the bottom of a long scroll, and feedback was "what am I supposed to do with this list?"
Iteration 3 — tabbed layout, selections at top ✓
Selection boxes anchored at the top. Tabs below to switch between lists. Analyse button beside the selections — always visible, never buried.
What the final design does and why
  • Two selection boxes sit at the top — one for the authority document, one for the internal document Both visible simultaneously so users can verify their pair before running the analysis
  • Each box has placeholder text explaining what goes there before anything is selected "Add from Internal Documents list / It will be shown here" — reduces confusion without a separate onboarding flow
  • Each selected document shows an X to remove it independently Users can swap one document without losing the other — no starting over
  • The document list uses tabs to switch between Authority Documents and Internal Documents Single screen, no navigation — users browse and select without leaving the page
  • The Analyse Documents button sits beside the selection boxes, not at the bottom of the list Always visible regardless of how far the user scrolls through the document list
  • The analysis history shows a "Processing..." state inline with a progress bar Users can track progress without leaving the page — no separate loading screen
The label "Authority Document" came directly from card sorting research — users consistently used this term over alternatives like "regulatory document" or "external document."
What I'd do differently — and what's already planned

The current design places the Analysis History tab at the same level as the document selection tabs. That created a hierarchy problem — analysis history has nothing to do with document selection, but it sits alongside selection tools as if it does. A user encountering it for the first time could reasonably wonder what it's doing there.

The next version addresses this directly. The main page becomes the analysis history — a list of all previous and in-progress analyses. Starting a new analysis triggers a modal where users search and select their two documents simultaneously. This came from a conversation with the developers, who flagged that a search-while-viewing-list interaction was technically feasible. I wouldn't have known to propose it without that input — and it unlocked a significantly better flow than what I'd been assuming was possible.

Next version — in development
Analysis history as the main view → "Add New Analysis" button → modal with live search across both document lists simultaneously. Cleaner hierarchy, faster selection, no tab confusion.
What this taught me

"When there are no references to follow, the constraints you set for yourself become the design. I didn't find the answer by looking at other products — I found it by being clear about what the user needed to see and do at every moment on the screen. And the best improvement to this design didn't come from me — it came from listening to the developers."
